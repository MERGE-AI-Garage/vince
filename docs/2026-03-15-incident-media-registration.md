# Incident: AI-Generated Images Invisible in Media Library
**Date:** March 15, 2026
**Severity:** High — all images generated after ~midnight went unregistered
**Duration:** ~3 hours (02:00–05:00 UTC)
**Resolved:** Yes

---

## What Broke

Images generated by `generate-creative-image` and `brand-prompt-agent` (headshot scenes) were completing successfully — Gemini returned images, storage uploads succeeded, `creative_studio_generations` records were created — but nothing appeared in the media library sidebar. The user could not find recently-generated images under AI Generated → Studio or AI Generated → Campaigns.

Separately, `generate-creative-package` was returning 401 for all calls (blocking campaign creation through `brand-prompt-agent`).

---

## Root Cause

### Issue 1: Media inserts failing silently (primary)

Both `generate-creative-image/index.ts` and `brand-prompt-agent/index.ts` were updated on 2026-03-14 to write SynthID provenance fields (`synthid_detected`, `synthid_confidence`, `synthid_generated_by`, `synthid_details`) into the `media` table insert. The code was deployed without a corresponding schema migration.

PostgREST rejects inserts that reference columns that don't exist, returning a column-not-found error. The calling code does not check the error from the media insert (by design — media registration is non-blocking, so failures are silent). The result: images landed in Supabase storage and `creative_studio_generations`, but never in the `media` table.

Evidence:
- `creative_studio_generations` had 8 completed records from today
- `media` table had zero entries from today
- `/Creative Studio/Demo_User` folder was created (folder insert succeeded) but contained 0 images
- `media` table schema confirmed: no `synthid_*` columns

### Issue 2: `generate-creative-package` returning 401

A redeployment (version 29) reset `verify_jwt` to the Supabase default (`true`), causing platform-level auth rejection before function code ran. Same root cause as the March 13 incident.

### Issue 3: Headshot scenes lacked `folder_id`

The `generateHeadshotScene()` media insert in `brand-prompt-agent` did not set `folder_id`. Images would have landed in the media table root (null folder), invisible in the sidebar folder tree even if the insert had succeeded.

## Fix Applied

**1. Schema migration** — added 4 missing columns to the `media` table:

```sql
ALTER TABLE public.media
  ADD COLUMN IF NOT EXISTS synthid_detected boolean DEFAULT false,
  ADD COLUMN IF NOT EXISTS synthid_confidence numeric DEFAULT NULL,
  ADD COLUMN IF NOT EXISTS synthid_generated_by text DEFAULT NULL,
  ADD COLUMN IF NOT EXISTS synthid_details text DEFAULT NULL;
```

Migration: `supabase/migrations/20260315_add_synthid_columns_to_media.sql`

**2. Backfill** — inserted media records for all 7 stranded images using their existing `output_urls` from `creative_studio_generations`. Headshot scenes → `/AI Generated/Studio`; text_to_image and multi_turn_edit → `/Creative Studio/Demo_User`.

**3. Fixed `folder_id` in headshot scene media insert** — `brand-prompt-agent/index.ts` now looks up `/AI Generated/Studio` folder before inserting headshot scene media records.

**4. Redeployed `generate-creative-package` and `brand-prompt-agent`** with `--no-verify-jwt`:

```bash
npx supabase functions deploy generate-creative-package --no-verify-jwt
npx supabase functions deploy brand-prompt-agent --no-verify-jwt
```

---

## Prevention

The SynthID columns were added to code (2026-03-14) before the schema migration was applied. Going forward: **code and migration must ship together**. If a function insert references new columns, the migration must be applied (or the column must be spread into an existing JSONB field like `ai_provenance`) before deploying the function.

See also: [2026-03-13-incident-edge-function-auth.md](2026-03-13-incident-edge-function-auth.md) for the `--no-verify-jwt` deployment rule.
